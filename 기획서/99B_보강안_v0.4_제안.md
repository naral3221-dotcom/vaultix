### G2 / D-10. ComfyUI SPOF 대응 — 가용 SLA + 휴가 모드

**배경**: 데스크탑 ComfyUI는 JH 부재(본업 출장, 정전, 윈도우 강제 재부팅 등) 시 자동 폴백이 발동하는데, 그 비용·품질 부담이 누적되면 곤란.

#### 가용 시간대 SLA (자체 약속)

| 시간대 | 보장 | 비고 |
|--------|:---:|------|
| 평일 22:00 ~ 08:00 KST | ★★★ 보장 | 야간 배치 메인 시간대 |
| 주말 24h | ★★ 가능한 한 | 가족 일정 시 휴가 모드 |
| 평일 08:00 ~ 22:00 | ★ 부재 가능 | 본업 시간 |

**SLA를 어겨도 자동 폴백이 처리하지만**, 누적 폴백 비용(비상 API)이 한도 초과로 가지 않도록 다음을 운영.

#### 휴가 모드 (Vacation Mode)

`/admin/vacation`에서 토글:
- 자산 자동 발행 일시 중지 (인박스만 누적)
- 큐레이션 자동 임계 상향(0.5 → 0.65, 사람 손 없이도 어느 정도 품질 유지)
- 외부 API 호출 비활성화 (Nanobanana도 일시 정지 옵션)
- Telegram 알림은 P0(서비스 다운)만 유지

복귀 시 토글 해제 → 누적된 인박스 일괄 처리.

#### 원격 제어 옵션 (선택)

- **Tailscale Subnet Router**로 데스크탑 외 IP에서도 데스크탑 ComfyUI 접근(이미 가능)
- **Wake-on-LAN**: 본업 출장 중 데스크탑이 절전 진입했을 때 원격 wake. 공유기 설정 + WoL 패킷 송출 스크립트로 구현. 우선순위 낮음 — Phase 4+에서 검토.
- **항시 가동 정책**: 데스크탑 전기료 추산 (R9700 idle ~50W × 24h × 30일 ≈ 36kWh/월 ≈ 6,000~8,000원). 부재 중에도 켜둔다면 명시적 결정 필요.

### G3 / T-11. RTO/RPO + WAL 아카이빙 (바이브 코더용 설명 포함)

**용어 풀이**:
- **RTO(Recovery Time Objective)** = "장애가 났을 때 몇 분/시간 안에 다시 살려야 하는가"
- **RPO(Recovery Point Objective)** = "장애가 났을 때 최대 몇 분/시간치 데이터까지는 잃어버려도 되는가"

v0.3의 백업 정책은 "일 1회 pg_dump + 주 1회 복원 테스트"라 RPO가 사실상 24시간이야. 하루치 가입자·다운로드 이력이 통째로 날아갈 수 있다는 뜻. 1인 운영이라도 사용자 신뢰엔 치명적.

#### 권장 설정

| 지표 | 목표 | 의미 |
|------|:---:|------|
| **RTO** | 4시간 | 장애 발생 후 4시간 내 복구 |
| **RPO** | 1시간 | 최대 1시간치 데이터 손실 허용 |

#### 구현 방법 (PostgreSQL WAL Archiving)

WAL(Write-Ahead Log)은 PG가 모든 변경을 기록하는 작은 로그 파일이야. 이걸 1분~5분 단위로 외부 스토리지(R2)에 송출하면 RPO를 분 단위까지 줄일 수 있음.

**도구 추천**: [pgBackRest](https://pgbackrest.org/) (성숙도·문서 우수) 또는 [wal-g](https://github.com/wal-g/wal-g)

```yaml
# docker-compose 추가 (Phase 0)
pgbackrest:
  image: pgbackrest/pgbackrest:latest
  environment:
    - PGBACKREST_REPO1_TYPE=s3
    - PGBACKREST_REPO1_S3_BUCKET=${R2_BUCKET}
    - PGBACKREST_REPO1_S3_ENDPOINT=${R2_ENDPOINT}
  volumes:
    - postgres_data:/var/lib/postgresql/data
  command: archive-push
```

- 풀 백업: 주 1회
- 증분 백업: 일 1회
- WAL 송출: 5분 간격 (자동)

#### 자산 파일 손상 별도 대응

DB와 별개로 `/data/assets/` 디렉토리도 손상 가능. 권장:
- **체크섬 DB**: 자산 발행 시 SHA-256을 `assets.checksum` 컬럼에 저장
- **주 1회 검증 작업**: 모든 자산 SHA-256 재계산 → DB와 다르면 알림
- **Glacier 티어 백업**: R2 또는 Backblaze B2로 일 1회 자산 디렉토리 동기화 (rclone)

#### 복구 리허설 (필수)

분기 1회는 "DB 전체 날렸다 가정 → 복원" 시뮬레이션 실행. 이걸 안 해보면 진짜 장애 시 절차가 어떻게 꼬이는지 모름. A6 운영 런북에 절차 추가.

#### 액션 아이템

- [ ] Phase 0 W7에 pgBackRest 또는 wal-g 컨테이너 추가
- [ ] R2 버킷 생성·자격증명 환경변수 등록
- [ ] `assets.checksum` 컬럼을 A1 DDL에 추가

### G4 / B-09. 신고/삭제 페이지 (Notice & Takedown)

**배경**: AI 생성물 저작권/초상권 침해 신고는 시간 문제일 뿐 반드시 들어옴. 응답 절차가 없으면 AdSense·OAuth 정책 위반 → 계정 정지 위험.

#### `/report` 페이지 설계

- 다국어 노출 (한·영·일)
- 양식 필드:
  - 신고자 정보 (이름·이메일·연락처)
  - 신고 대상 자산 URL (자동 검증 — 우리 도메인 자산만 허용)
  - 신고 유형 (저작권 / 초상권 / 상표권 / 명예훼손 / 기타)
  - 권리 보유 증빙 (파일 업로드 — 5MB까지)
  - 진술서 (체크박스 — "선의의 신고이며 허위 시 법적 책임" 동의)
- 제출 시:
  - DB 저장 (`takedown_requests` 테이블)
  - JH Telegram 즉시 알림
  - 신고자에게 접수 확인 메일 (티켓 번호 포함)

#### 처리 SLA

| 유형 | 응답 시간 | 조치 |
|------|:---:|------|
| 명백한 침해 (육안 확인) | 24시간 | 즉시 비공개 + 검토 |
| 유사도 분쟁 | 영업일 3일 | JH 검토 후 결정 |
| 허위 신고 의심 | 영업일 7일 | 신고자에 추가 자료 요청 |

#### 차단 메커니즘

비공개 처리 시 추가 작업:
- 자산을 `status = 'taken_down'`으로 마킹 (URL은 410 Gone 반환)
- 해당 자산의 **프롬프트 + 시드 + 모델 조합**을 `blocked_recipes` DB에 추가
- 자동 발행 파이프라인이 매번 이 DB와 대조 → 같은 조합 재생성 차단

#### 액션 아이템

- [ ] `takedown_requests`, `blocked_recipes` 테이블을 A1 DDL에 추가
- [ ] `/report` 페이지를 Phase 1 W6(검증 단계)에 구현

### G5 / T-12. 자산 품질 메트릭 + 어드민 대시보드 ★★★

**배경**: 큐레이션 인박스(공개 전 검증)는 잘 설계됐는데, **공개 후 자산이 잘 됐는지** 측정하는 메트릭이 없음. 이게 없으면 큐레이션 임계값 조정·카테고리 우선순위·생성 워크플로우 개선의 근거가 사라짐.

#### 추적 메트릭 (자산 단위)

| 메트릭 | 정의 | 추적 위치 |
|--------|------|----------|
| `views` | 자산 상세 페이지 조회수 | `asset_metrics_daily` |
| `download_starts` | 다운로드 버튼 클릭 | 동일 |
| `downloads_completed` | 실제 파일 전송 완료 | 동일 |
| `time_to_first_download` | 발행→첫 다운로드까지 분 | `assets.first_download_at` |
| `ctr` | downloads / views | 계산 컬럼 |
| `share_rate` | 공유 클릭 / views | 동일 |
| `report_count` | 신고 횟수 | 동일 |

#### 추적 메트릭 (집계 단위)

- **카테고리별**: 평균 CTR, 다운로드 합계, 신고 비율
- **태그별**: 같은 항목
- **모델/워크플로우별**: 같은 항목 (어떤 워크플로우가 잘 먹히는지)
- **시간대별**: 발행 시간 vs 다운로드 시간 패턴
- **언어별**: ko/en/ja 사용자 다운로드 비율

#### 어드민 대시보드 — 메인 화면 위젯

**좌측 컬럼 (실시간 운영)**

### I6 / D-11. 다국어 우선순위 한 → 일 → 영 재검토

**배경**: 현재 D-7은 한·영·일 동시. 그러나 일본 시장이 영어권보다 우리에게 유리할 수 있음.

#### 일본 우선의 근거

- 일본 직장인 PPT/Word 수요는 한국과 거의 동일하게 큼
- 일본 AI 무료 자산 사이트 경쟁이 영어권보다 덜함
- 한국·일본 디자인 정서가 미국·유럽보다 가까움 (톤 조정 부담 ↓)
- 일본 검색 시장에서 한국 사이트는 신선한 진입자
- DeepL 한→일 품질이 한→영보다 안정적

#### 영어 후순위의 근거

- 영어권은 Pinterest/Reddit/Pixabay 등 강력한 선발주자
- 영어 SEO 경쟁이 미친 수준 (한국·일본 대비 10배 비용)
- 톤·문화 차이가 큼 (직장인 PPT 스타일 자체가 다름)
- "ESL 영어"로 보이는 어색함 위험

#### 제안

- D-7 변경: **한국어(런칭) → 일본어(Phase 2 W3) → 영어(Phase 4)**
- DeepL 무료 한도(50만자/월)는 ko→ja만으로도 충분
- 영어는 Phase 4에서 "일본어 안정화 후 추가"

### D.1 Nanobanana 및 GPT Image 2.0

**확정**: JH가 **Anthropic, OpenAI, Google AI, Z.AI 4개사 API 키 무제한 보유**.

### D.3 다국어 우선순위 한→일→영
