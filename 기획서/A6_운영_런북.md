# A6 — 운영 런북 (Runbook)

> **목적**: Phase 1 가동 직후부터 일상적으로 발생하는 운영 작업과 사고 대응 절차를 한 문서에 모음. 새벽에 알림이 울려도 졸린 눈으로 따라할 수 있을 만큼 구체적으로 작성.
> **작성일**: 2026-04-27
> **선행**: Phase 0 §G 백업 / A3 §9 ComfyUI 실패 매트릭스

---

## 0. 사용 안내

- 본 런북은 **살아있는 문서**: 새 사고가 발생할 때마다 SOP 추가/갱신
- 알림 채널: Telegram (@vaultix_alerts 봇)
- 우선순위 표기: P0(즉시 대응) / P1(2시간 내) / P2(다음 영업일) / P3(주간 검토)
- 모든 SOP는 `[감지 → 진단 → 조치 → 검증 → 기록]` 5단계 구조

## 2. 백업 / 복원

### 2.1 백업 자동화 (이미 가동)

```bash
# /opt/vaultix/scripts/backup.sh — Phase 0 §G에서 작성
# crontab: 0 4 * * * /opt/vaultix/scripts/backup.sh >> /var/log/vaultix/backup.log 2>&1
```

### 2.2 백업 검증 (월간)

```bash
# /opt/vaultix/scripts/check_backup.sh
set -euo pipefail
LATEST=$(ls -t /var/lib/vaultix/backup/vaultix-*.dump | head -1)
SIZE=$(stat -c%s "$LATEST")
if [ "$SIZE" -lt 10000000 ]; then  # 10MB 미만이면 의심
  curl -X POST -d "chat_id=$TG_CHAT&text=백업 파일 크기 비정상: $LATEST ($SIZE bytes)" \
    "https://api.telegram.org/bot$TG_TOKEN/sendMessage"
  exit 1
fi
echo "OK: $LATEST ($SIZE bytes)"
```

### 2.3 복원 절차 (분기 1회 + 사고 시)

```bash
# 1. 별도 컨테이너에 복원 (운영 PG 건드리지 않음)
docker run -d --name vaultix-pg-restore --network vaultix_internal \
  -e POSTGRES_PASSWORD=temp -e POSTGRES_DB=vaultix_restore \
  postgres:17-alpine

# 2. 백업 파일 복사
docker cp /var/lib/vaultix/backup/vaultix-LATEST.dump vaultix-pg-restore:/tmp/

# 3. 복원
docker exec vaultix-pg-restore pg_restore -U postgres -d vaultix_restore /tmp/vaultix-LATEST.dump

# 4. 검증
docker exec vaultix-pg-restore psql -U postgres -d vaultix_restore -c "
  SELECT 'users' AS t, COUNT(*) FROM users
  UNION ALL
  SELECT 'assets', COUNT(*) FROM assets
  UNION ALL
  SELECT 'downloads', COUNT(*) FROM downloads;
"

# 5. 운영 DB와 행 수 비교 (텔레그램 보고)
# 6. 검증 완료 후 임시 컨테이너 제거
docker rm -f vaultix-pg-restore
```

## 3. 사고 대응 SOP

### 3.1 SOP-01 — 사이트 다운 (P0)

**감지**:
- Uptime Kuma → "vaultix.com is DOWN" 텔레그램
- 또는 사용자 제보

**진단** (각 단계 1분 이내):
```bash
# 1. SSH 접속 가능?
ssh openclaw@main-hub
# Tailscale로도 시도: ssh openclaw@100.116.156.37

# 2. nginx 상태
sudo systemctl status nginx
sudo nginx -t

# 3. 컨테이너 상태
docker ps -a | grep vaultix
docker compose -f /opt/vaultix/infra/docker-compose.yml ps

# 4. 디스크/메모리
df -h
free -h

# 5. 시스템 로그
sudo journalctl -xe -n 50
```

**조치 (시나리오별)**:
- nginx config 오류 → `sudo nginx -t` 출력 보고 수정 → `sudo systemctl reload nginx`
- 컨테이너 OOM → `docker logs vaultix-api --tail 200` 확인 → `docker compose restart`
- 디스크 가득 → §3.6 참조
- 데이터 손상 → §2.3 복원 절차

**검증**:
- 외부에서 https://vaultix.com 200 응답
- Uptime Kuma 모든 모니터 UP

**기록**:
- Phase 1~2: 텔레그램 채팅에 시간·증상·원인·조치 기록
- Phase 3+: incident.md에 사후 회고

### 3.2 SOP-02 — 데스크탑 ComfyUI 다운 (P1)

**감지**:
- 텔레그램: "ComfyUI down (3 consecutive checks failed)"

**진단**:
```bash
# VPS에서
curl -m 5 http://100.x.x.x:8188/system_stats   # 응답 없으면 데스크탑 또는 Tailscale 문제

# 데스크탑(직접) 확인
# 1. ComfyUI 콘솔 창 살펴보기 (에러 있는지)
# 2. Tailscale 클라이언트 상태 (트레이 아이콘)
# 3. ComfyUI 프로세스 작업관리자에서 확인
```

**조치**:
- ComfyUI 프로세스 죽음 → `run_comfyui_vaultix.bat` 재실행
- Tailscale 끊김 → 트레이에서 "Connect" 클릭
- GPU 드라이버 문제 → 재부팅
- 데스크탑 자체 다운 → 켜고 자동 시작 대기 (작업 스케줄러 등록되어 있음)

**검증**:
- VPS에서 `curl http://100.x.x.x:8188/system_stats` 200
- Redis: `redis-cli get health:comfy` → "up"
- 다음 5분 자동 헬스체크 통과

**자동 회복**:
- 그동안 발행은 fallback 큐로 자동 전환됨 (A3 §4)
- 외부 API 비용 약간 ↑, 캡 미달이면 무관

**기록**: 다운 빈도 추이가 주 3회+ 도달하면 데스크탑 안정화 작업 (W용 패치, 메모리, 드라이버)

### 3.3 SOP-03 — 외부 API 비용 폭증 (P1)

**감지**:
- 텔레그램: "외부 모델 호출량 평소 대비 5배 증가" 또는 "이미지 provider 실패율 20% 초과"

**진단**:
```bash
# DB에서 이번 달 외부 API 호출 수·일별 추이
docker exec vaultix-postgres psql -U $PG_USER -d vaultix -c "
SELECT
  DATE(created_at) AS day,
  generator,
  COUNT(*) AS calls
FROM generation_jobs
WHERE created_at >= DATE_TRUNC('month', NOW())
  AND generator IN ('nanobanana','gpt_image_2')
GROUP BY 1, 2
ORDER BY 1 DESC, 2;
"
```

**조치**:
- Nanobanana 실패율이 높으면 OpenAI `gpt-image-2` 폴백 비중과 실패 원인 확인
- 두 provider가 모두 불안정하면 `/admin/panic`으로 신규 생성만 일시 중단
- 도용 의심 호출량이면 API 키 회전 + 최근 `llm_call_log`/`generation_jobs` request_id 확인
- 장기: provider timeout, retry, 품질 임계값 조정

**검증**:
- /admin/queue/status에서 spend 값 정상
- 다음날 fallback 큐 사용량 50% 이상 감소

### 3.4 SOP-04 — DB 다운 / 응답 지연 (P0)

**감지**:
- API 5xx 응답 폭증 (Sentry)
- Uptime Kuma TCP 모니터 (PG 5440) DOWN

**진단**:
```bash
docker logs vaultix-postgres --tail 100
docker exec vaultix-postgres psql -U $PG_USER -d vaultix -c "SELECT NOW();"
# 응답 늦으면:
docker exec vaultix-postgres psql -U $PG_USER -d vaultix -c "
SELECT pid, now() - query_start AS dur, query
FROM pg_stat_activity
WHERE state = 'active' AND now() - query_start > interval '5 seconds'
ORDER BY dur DESC LIMIT 10;
"
```

**조치**:
- 컨테이너 다운 → `docker compose restart postgres`
- 슬로우 쿼리 → 인덱스 점검 (A1 §13), 필요 시 EXPLAIN ANALYZE
- 디스크 가득 → §3.6
- 메모리 압박 → shared_buffers 조정 (postgresql.conf) + 재시작

**검증**:
- API /healthz 200
- 다운로드 정상

**기록**: 슬로우 쿼리는 별도 backlog (PR로 인덱스 추가)

### 3.5 SOP-05 — 큐레이션 인박스 정체 (P2)

**감지**:
- 텔레그램: "인박스 100건 초과"
- 또는 매일 09:00 자동 알림

**진단**:
```sql
SELECT
  asset_type,
  COUNT(*) AS waiting,
  MIN(created_at) AS oldest
FROM assets
WHERE status = 'inbox'
GROUP BY asset_type;
```

**조치 (옵션)**:
1. **30분 집중 처리**: 키보드만으로 처리, 점수 높은 것부터
2. **점수 임계 일시 상향**: `min_score=0.7` 필터로 시간 절감
3. **자동 승인 정책 임시 활성화** (Phase 4 이후): `AUTO_APPROVE_THRESHOLD=0.85` 적용
4. **발행 페이스 일시 감속**: enqueue_daily_batch의 target_count를 25로 낮춤 (이번 주만)

**검증**: 인박스 < 30 도달

**기록**: 정체가 월 2회+ 발생하면 발행 페이스 영구 조정 또는 자동 승인 정책 검토

### 3.6 SOP-06 — 디스크 가득 (P0)

**감지**:
- 텔레그램: "/var 사용량 85%+"
- df -h 직접 확인

**진단**:
```bash
df -h
du -sh /var/lib/vaultix/* | sort -h
du -sh /var/log/vaultix/*

# 가장 큰 컨테이너
docker system df -v
```

**조치 (우선순위 순)**:
```bash
# 1. 도커 prune (안전)
docker system prune -f --volumes=false

# 2. 오래된 도커 이미지 (latest 외)
docker images | grep vaultix | grep -v latest | awk '{print $3}' | xargs docker rmi

# 3. 오래된 백업 dump 삭제 (gdrive에 있음)
find /var/lib/vaultix/backup/ -name "vaultix-*.dump" -mtime +14 -delete

# 4. 30일 이상 trash 자산 삭제
find /var/lib/vaultix/assets/trash/ -mtime +30 -delete

# 5. journald 로그 정리
sudo journalctl --vacuum-time=14d
sudo journalctl --vacuum-size=2G

# 6. nginx access log
sudo find /var/log/nginx/ -name "*.gz" -mtime +30 -delete
```

**대안 (R2 미이전 시)**:
- 50GB 이상이면 즉시 R2 마이그레이션 시작 (Phase 4 §W6)
- 단기 응급: published 자산을 gdrive 마운트(/mnt/gdrive)로 이동 + nginx alias 변경

**검증**: df -h, 디스크 < 70%

### 3.7 SOP-07 — 메일 도달 실패 (P1)

**감지**:
- 사용자: "인증 메일 못 받았어요"
- Resend 대시보드: bounce·complaint 증가

**진단**:
1. Resend 대시보드 → 해당 메일 status 확인 (delivered / bounced / complained)
2. mail-tester.com 점수 확인 (자체 도메인 보낼 메일)
3. SPF/DKIM/DMARC 레코드 점검 (`dig TXT <domain>`)

**조치**:
- bounced (영구) → 사용자에게 다른 이메일 사용 안내
- bounced (일시) → resend-verification 5분 후 재시도
- complaint → 해당 사용자 차단 + 이유 분석
- mail-tester < 8/10 → SPF/DKIM/DMARC 보강

**검증**: 5종 인박스(Gmail/Naver/Kakao/Daum/Hotmail) 도착 검증

**기록**: bounce율 > 3% 도달 시 Resend 계정 위험 (Resend 정책)

### 3.8 SOP-08 — 라이선스 분쟁 신고 도착 (P1)

**감지**:
- contact@<domain> 메일: "이 이미지가 우리 회사 로고와 유사합니다"

**진단**:
1. 신고 자산 확인 (asset_id)
2. 신고 사유와 자산 비교 (실제 유사한가?)
3. 신고자의 권리 명시 (상표 등록증, 저작권 등)

**조치**:
1. **즉시**: 자산 unpublish (`/admin/assets/{id}/archive`) — 분쟁 중 노출 X
2. 신고자에게 24시간 내 답변
3. 분쟁이 명백히 정당하면 → 영구 삭제 + 다운로드한 사용자에게 안내 (선택)
4. 애매하면 → 무료 법률상담 또는 관련 자료 정리 후 신중히 답변

**검증**: 자산 unpublish 후 사이트에 표시되지 않음

**기록**: incidents.md에 사례 누적 — 패턴 분석 (특정 카테고리에서 자주 발생하면 프롬프트 템플릿 수정)

### 3.9 SOP-09 — 보안 의심 활동 (P0)

**감지**:
- fail2ban 알림 (반복 SSH 실패)
- 비정상 다운로드 패턴 (1초에 10건 등)
- 신규 가입 1분에 50건 (봇)

**진단**:
```bash
# fail2ban 상태
sudo fail2ban-client status sshd

# 의심 IP의 활동
docker exec vaultix-postgres psql -U $PG_USER -d vaultix -c "
SELECT * FROM downloads
WHERE ip = 'X.X.X.X'
ORDER BY downloaded_at DESC LIMIT 50;
"

# 신규 가입 추이
docker exec vaultix-postgres psql -U $PG_USER -d vaultix -c "
SELECT DATE_TRUNC('minute', created_at), COUNT(*)
FROM users
WHERE created_at > NOW() - INTERVAL '1 hour'
GROUP BY 1 ORDER BY 1 DESC;
"
```

**조치**:
- ufw에 IP 차단: `sudo ufw insert 1 deny from X.X.X.X`
- Cloudflare WAF 룰 (Phase 2+에서 Cloudflare 도입 시)
- 봇 가입 → 해당 사용자들 일괄 suspend + 일회용 메일 도메인 추가
- Turnstile 임계 강화 (점수 0.7+ 만 통과)

**검증**: fail2ban 차단 목록, 비정상 패턴 종료

**기록**: 차단 IP·이유 별도 노트, 패턴 발견 시 자동 차단 룰 추가

### 3.10 SOP-10 — 데이터 유출 의심 (P0, 가장 중요)

**감지**:
- DB dump 파일이 검색엔진에 노출
- 시크릿(.env, API 키) 공개 git에 push
- 사용자 비밀번호 평문 노출
- Cloudflare R2 버킷 public 노출

**즉시 조치**:
1. 노출된 시크릿 즉시 무효화·재발급
   - `.env` 변경, 모든 API 키 회전
   - DB password 변경, 세션 전부 무효화 (`DELETE FROM sessions`)
2. Sentry/감사 로그로 영향 범위 파악
3. 영향받은 사용자 식별 + 24~48시간 내 통지 (한국 개인정보 보호법 준수)
4. 개인정보보호위원회 신고 검토 (1,000명 이상 영향 시 의무)
5. incident.md 작성 + 외부 공지 (필요 시)

**예방**:
- pre-commit hook의 secret scan
- Renovate에 의해 의존성 업데이트
- CI에 trivy/grype 스캔
- 정기 보안 점검
