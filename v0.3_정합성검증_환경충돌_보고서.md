## 2. v0.3 가정 ↔ 운영 환경(main-hub) — ⚠️ 4건의 충돌/조정 필요

### 충돌 #1 — GPU 없음 (가장 큰 변경)

**v0.3 가정** (§2.1, §10.1, §10.2, §5.5)
- 이미지 생성: SDXL/Flux 로컬, image-worker (GPU)
- LLM: Llama 3.1 70B 로컬 운영 가정
- "00:00~08:00 이미지 배치, 08:00~12:00 LLM 메타데이터" 식 GPU 시간 분리

**실제 환경**: GPU 없음 (QEMU virtual VGA만)

**의미**:
- SDXL/Flux 로컬 추론은 사실상 불가 (CPU 추론은 1장당 수 분~수십 분)
- 70B 로컬 LLM 무리. 7B~13B 양자화 모델만 실용적
- v0.3의 "일 이미지 50~100장" 목표는 외부 API 또는 다른 전략 필요

**대안 옵션** (Phase 0~1에서 결정):
| 옵션 | 비용 | 품질 | 운영 부담 |
|------|------|------|----------|
| A. 외부 API (Replicate/fal.ai/Together) 일 50장 | 일 $1~3, 월 $30~90 | SDXL/Flux 동급 | 낮음 |
| B. SD 1.5 + LCM/Turbo CPU 추론 | 0 | SDXL보다 떨어짐, 일러스트 한정 가능 | 보통 (생성 속도 분 단위) |
| C. 외부 API + 로컬 후처리(업스케일/배경제거) 하이브리드 | 월 $20~50 | 우수 | 보통 |
| D. GPU VPS 별도 임대 (Vast.ai 등 spot) | 시간당 $0.2~0.5, 월 $30~80 | 최상 | 높음(스팟 인스턴스 관리) |

→ **JH 결정 필요 (D-1)**

### 충돌 #2 — 시스템 nginx가 80/443 점유

**v0.3 가정** (§10.1): Nginx Reverse Proxy를 도커 스택의 일부로 가정

**실제 환경**: 시스템 `nginx.service`(apt)가 80/443 점유. 다른 25개 컨테이너도 모두 이 nginx의 server block으로 노출 중

**조정안**:
- 새 컨테이너는 호스트 포트 노출하지 않거나 내부 포트만 (예: 127.0.0.1:8301)
- `/etc/nginx/sites-available/imghub.conf` 추가하고 `proxy_pass http://127.0.0.1:8301;`
- Docker 컨테이너에 `nginx-proxy` 또는 `Caddy` 새로 띄우지 말 것
- TLS는 시스템 certbot 사용 (이미 운영 중일 가능성 — 점검 필요)

→ **이건 자동 결정 가능. JH 컨펌만 받으면 됨 (D-2)**

### 충돌 #3 — PostgreSQL 인스턴스 8개가 이미 운영 중

**v0.3 가정** (§10.2): PostgreSQL 16 신규 컨테이너

**실제 환경**: 5433, 5435~5439 점유. 기존 PG 컨테이너에 DB만 추가하는 옵션도 있음

**옵션**:
| 옵션 | 장점 | 단점 |
|------|------|------|
| A. 새 PG 컨테이너 (포트 5440) | 완전 격리, 백업·튜닝 독립 | 메모리 약간 더 사용 (300~500MB) |
| B. 기존 컨테이너에 DB 추가 | 자원 절약 | 다른 프로젝트와 결합, 마이그레이션·백업 시 같이 영향 |

추천: **A (새 컨테이너 5440)** — RAM 47GB 충분하고, AI 컨텐츠 허브는 향후 트래픽 늘면 DB 분리가 어차피 필요.

→ **추천대로 가도 되는지 JH 컨펌 (D-3)**

### 충돌 #4 — Ollama 설치 (v0.3는 가정, 환경엔 없음)

**v0.3 가정** (§5.4): Ollama 1차 LLM

**실제 환경**: Ollama 미설치. 그리고 GPU 없음 → CPU 추론

**의미**: 7B~13B 양자화 모델만 실용 (Qwen 2.5 7B Q4_K_M, Llama 3.1 8B Q4 등)
- 메타데이터 생성: CPU로도 충분 (텍스트 짧고 배치 처리 가능)
- 블로그 초안: CPU는 느림 → 외부 API(Claude/GPT) 권장

**Phase 0 액션**: Ollama 설치 + Qwen 2.5 7B Q4 받아서 CPU 메타데이터 생성 벤치 (장당 처리 시간 측정)

→ **별도 결정 불필요. Phase 0 작업 항목으로 포함**

## 3. 추가로 확인된 환경 자산 (v0.3에 반영해야 할 +α)

| 자산 | v0.3에 없던 활용 |
|------|-----------------|
| gdrive 5TB rclone 마운트 | 백업·아카이브 즉시 활용. R2 이전 전까지도 외부 백업 OK |
| Tailscale 운영 중 | 관리 트래픽(어드민, DB 직접 접속, Plausible 대시보드)을 100.x.x.x로 분리 가능 → 보안 ↑ |
| n8n 운영 중 | 토픽 발굴·뉴스레터 발송·알림 워크플로우를 n8n으로 빌드하면 코드 절감 |
| cloudbeaver 8201 운영 중 | 신규 PG도 cloudbeaver에 등록해서 GUI 관리 가능. 별도 DB 툴 불필요 |
| uv 0.10.9 설치 | poetry 대신 uv로 통일 (v0.3는 비명시) |

→ Phase 0 기획서에 위 자산을 활용하는 방향으로 반영하겠음.
